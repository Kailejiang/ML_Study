# LLM 核心概念全链路梳理解释（从基础到Agent智能体）

## 一、基础核心单元：LLM的文本处理与载体
这是所有LLM应用的底层基石，解决「模型怎么读文本、能读多少文本」的问题。

### 1. LLM（Large Language Model，大语言模型）
- **定义**：基于Transformer Decoder架构、通过海量文本预训练而成的**概率语言模型**，核心能力是**预测下一个Token**，具备理解、生成、推理、对话等通用语言能力。
- **本质**：LLM是「大脑」，本身无记忆、无联网、无计算能力，仅能处理文本Token。
- **代表**：GPT系列、Llama、文心一言、通义千问等。

### 2. Token（令牌）
- **定义**：LLM**处理文本的最小原子单位**，不是汉字/英文单词，而是**子词单元**（由Tokenizer拆分）。
- **特点**：
  - 英文：1 Token ≈ 4字符 / 0.75个单词；
  - 中文：1 Token ≈ 1个汉字/词；
- **作用**：LLM不直接识别文字，只识别Token对应的数字ID，所有计算、计数都以Token为单位。

### 3. Tokenizer（分词器）
- **定义**：LLM的**文本编码器/解码器**，负责「人类文本 ↔ Token ID」的双向转换。
- **核心功能**：
  1. 编码：把用户输入的文字拆分成Token，转为数字ID输入模型；
  2. 解码：把模型输出的Token ID还原为人类可读文字；
- **关键**：不同模型有专属Tokenizer（如GPT、Llama），拆分规则不同，直接影响Token计数和模型理解。

### 4. Context（上下文）
- **定义**：**LLM在当前交互中能「看到」的所有文本信息总和**，是模型的「临时记忆」。
- **包含内容**：系统提示、用户提问、历史对话、工具返回结果、模型自身输出等。
- **作用**：模型所有推理、生成都**仅基于上下文**，无外部记忆，上下文决定模型的决策依据。

### 5. Context Window（上下文窗口）
- **定义**：LLM**能处理的最大Token数量上限**（硬限制），是模型的「记忆容量」。
- **例子**：GPT-4o 128K Token、Llama 3 70B 128K Token；
- **后果**：超过窗口大小的文本会被**截断/遗忘**，长对话、长文档推理会失效；
- **关联**：Context是窗口内的实际内容，Context Window是窗口的容量上限。

## 二、交互控制层：引导LLM输出的核心逻辑
解决「怎么让LLM精准听懂指令、按要求输出」的问题，是LLM交互的核心。

### 1. Prompt（提示词）
- **定义**：**直接输入给LLM的全部文本内容**，是人类与LLM交互的唯一接口。
- **构成**：系统提示 + 用户提示 + 历史对话 + 示例等所有上下文文本。
- **本质**：Prompt = 给LLM的「任务说明书」，模型完全基于Prompt生成输出。

### 2. Prompt Engineering（提示工程）
- **定义**：**设计、优化Prompt的方法论**，用最小成本让LLM输出精准、合规、高质量结果。
- **核心技巧**：清晰指令、角色设定、少样本示例、格式约束、思维链（CoT）、约束规则等；
- **作用**：不修改模型参数，仅通过文字引导放大LLM能力，是LLM应用的基础技能。

### 3. System Prompt（系统提示）
- **定义**：**给LLM的底层设定、角色、规则与约束**，优先级最高，贯穿整个对话。
- **内容**：设定身份（如「你是Python工程师」）、输出格式、禁忌规则、行为规范；
- **特点**：用户不可见，固定在上下文顶部，决定LLM的基础行为模式。

### 4. User Prompt（用户提示）
- **定义**：**用户的实际需求、问题、指令**，是单次交互的核心输入。
- **特点**：动态变化，每次对话更新，是LLM需要解决的具体任务。

## 三、外部能力扩展：弥补LLM天生缺陷
LLM本身无实时信息、无计算、无操作能力，通过工具和协议扩展能力。

### 1. Tool（工具）
- **定义**：LLM调用的**外部功能插件**，用于弥补自身短板。
- **常见工具**：计算器、搜索引擎（联网）、代码解释器、数据库、API接口、文件读写、绘图工具等；
- **作用**：让LLM从「纯文本生成」变成「能做事、能交互、能获取真实信息」的实用工具。

### 2. MCP（Model Context Protocol，模型上下文协议）
- **定义**：**LLM与工具/环境交互的标准化协议**，统一工具调用的格式、参数、返回结果规范。
- **核心作用**：
  1. 规定LLM如何「请求调用工具」（如指定工具名、入参）；
  2. 规定工具如何「返回结果给LLM」；
  3. 让Agent、工具、模型之间的交互标准化，兼容多工具、多场景；
- **关联**：是LLM调用Tool的「通用语言」，无MCP则工具调用混乱不可控。

## 四、自主智能体系：Agent智能体（LLM的终极形态）
解决「LLM只能被动响应，无法自主解决复杂任务」的问题，是LLM从「工具」变「智能助手」的核心。

### 1. Agent（智能体）
- **定义**：以**LLM为大脑**，整合「记忆+规划+工具调用+反馈迭代」的**自主决策系统**。
- **核心组件**：LLM（推理）+ Context（记忆）+ Tool（能力）+ 决策范式（思考逻辑）；
- **特点**：不再被动生成文本，而是**自主感知任务→思考方案→调用工具→验证结果→完成目标**，可处理复杂多步任务。

### 2. ReAct（Reasoning + Acting，推理-行动范式）
- **定义**：Agent最基础、最通用的决策范式，**同步进行推理与行动**。
- **执行流程**：
  思考（Reasoning）→ 行动（Acting，调用Tool）→ 观察结果 → 再思考 → 再行动 → 完成任务；
- **特点**：边想边做、轻量灵活，适合简单/中等复杂度任务；
- **地位**：Agent的入门核心逻辑，几乎所有基础Agent都基于ReAct。

### 3. Plan and Execute（规划-执行范式）
- **定义**：Agent的**高级决策范式**，先全局规划，再分步执行，解决ReAct「短视、无全局观」的问题。
- **执行流程**：
  全局规划（拆解任务步骤）→ 按序执行每一步（调用Tool）→ 实时调整规划 → 最终完成；
- **特点**：全局视角、步骤可控，适合长流程、高复杂度任务（如数据分析、项目执行、代码开发）。

### 4. Agent Skill（智能体技能）
- **定义**：Agent封装的**模块化、可复用的能力单元**，是「Tool + 专用Prompt模板」的组合。
- **例子**：天气查询技能、Excel数据分析技能、代码编写技能、简历优化技能；
- **作用**：让Agent无需每次从头思考，直接调用预制技能，提升效率、降低错误率，是Agent的「能力库」。

## 五、全概念串联总结（完整执行链路）
把所有概念串成**用户发起任务 → LLM处理 → Agent自主完成**的完整流程，一眼看懂逻辑：

1. **用户输入需求**：发出User Prompt（实际问题）；
2. **文本编码**：Tokenizer将「System Prompt + User Prompt + 历史对话」转为**Token**；
3. **上下文加载**：所有Token装入**Context**，不超过**Context Window**上限；
4. **提示引导**：通过**Prompt Engineering**优化的Prompt，引导**LLM**理解任务；
5. **自主决策**：LLM作为Agent大脑，通过**ReAct/Plan and Execute**范式思考：
   - 是否需要调用**Tool**？
   - 调用哪个**Agent Skill**？
6. **工具交互**：通过**MCP**协议规范调用工具，获取外部结果；
7. **结果迭代**：工具返回结果加入**Context**，LLM继续推理，直到完成任务；
8. **文本解码**：Tokenizer将模型输出的Token还原为文字，返回给用户。

---

## 核心一句话总结
**LLM是大脑，Token是最小处理单元，Tokenizer是翻译官，Context Window是记忆容量，Prompt是指令，Prompt Engineering是指令优化，System+User Prompt构成指令主体，Tool是外挂能力，MCP是交互协议，Agent是带自主决策的LLM增强体，ReAct/Plan and Execute是决策逻辑，Agent Skill是模块化能力**，共同组成了现代大语言模型从文本处理到自主完成复杂任务的完整体系。
